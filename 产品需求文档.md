# AI 远程控制桌面系统（WebRTC + 控制码）— 产品需求文档（PRD）

**文档版本**：v1.1  
**更新说明**：采用「轻信令 + WebRTC P2P」架构，以控制码建立连接；**不使用 VNC 协议**；屏幕与控制走 WebRTC；在 MVP 与演进路线中明确局域网优先与后续 NAT/AI 扩展。**同一 Electron 安装包同时支持被控端与控制端**（模式切换，非两套独立程序）。

---

## 1. 项目概述

### 1.1 项目目标

构建基于 **WebRTC** 的跨平台远程桌面控制系统，实现：

- 通过「**控制码**」快速建立连接
- **局域网优先**连接（MVP 阶段）
- 支持后续**公网穿透**（STUN/TURN）扩展
- 支持**屏幕共享**与**鼠标、键盘**控制
- 支持**双向控制**能力（未来扩展）
- 支持 **AI 自动化控制**扩展（预留架构，与视觉规划闭环对接）
- **一套桌面程序**：同一安装包在**被控端（Agent）**与**控制端（Controller）**均可使用，通过**运行模式**切换或入口区分，无需分别安装两个应用

### 1.2 核心理念：轻服务器 + 强 P2P

| 原则 | 说明 |
|------|------|
| 信令轻量 | 信令服务器仅用于**设备匹配**与 **SDP/ICE** 信息交换 |
| 媒体与控制走 P2P | **视频流**与**控制数据**优先经 **WebRTC P2P** 传输，不经由服务器转发画面 |
| 连接顺序 | **优先局域网直连**；失败或跨网时再走 **STUN**；必要时 **TURN** 中继 |

### 1.3 产品定位（与 AI 能力结合）

在 **WebRTC 远程画面**（被控端采集、控制端渲染）之上，叠加 **屏幕理解 + AI 决策**，实现类人自动化：点击、输入、滚动等由结构化指令驱动；可选对步骤与整任务做视觉判定并输出报告。**不采用 VNC / RFB** 作为传输协议。

### 1.4 非目标（边界）

- 大规模并发多会话调度平台（若需要，单独立项）
- 复杂可视化用例编排器（首版可不做）
- 替代专业安全审计或合规认证工具
- 首版不要求完整企业级 TURN 自建（可列 Phase 3）

## 2. 系统架构设计

### 2.1 总体架构

```
        Signal Server（Node.js + WebSocket）
                 │
     ┌───────────┴───────────┐
     ▼                       ▼
 被控端（Agent Mode）      控制端（Controller Mode）
     │                       │
     └─── WebRTC P2P ────────┘
           （视频 + DataChannel）
```

### 2.2 技术组成

| 模块 | 技术 / 职责 |
|------|-------------|
| **统一桌面客户端** | **Electron**（Windows / macOS / Linux） |
| **被控端（Agent）** | 屏幕采集（Screen Capture）、WebRTC **发送**视频轨、经 **DataChannel** 接收控制指令并执行鼠标/键盘 |
| **控制端（Controller）** | WebRTC **接收**并渲染远程画面、经 **DataChannel** 发送输入事件、设备列表与连接管理 |
| **信令服务器（必须）** | **Node.js + WebSocket**：控制码绑定、**offer/answer** 与 **ICE candidate** 转发 |

### 2.3 统一桌面程序（被控与控制同一应用）

- **一个安装包、一个进程品牌**：对外交付 **单一 Electron 桌面应用**（Windows / macOS / Linux 按发布矩阵提供）。
- **双角色内置**：应用内同时包含 **Agent（被控）** 与 **Controller（控制）** 能力；用户在本机选择进入「被控模式」或「控制模式」，而非下载两个不同产品名安装包。
- **典型形态**（实现任选其一或组合）：启动向导或主界面提供模式选择；设置中切换模式（切换时需断开当前会话）；命令行参数 `--mode=agent|controller` 便于脚本与 CI。
- **MVP 不要求**同一时刻在本机同时充当双方建立自连；**要求**同一二进制/安装包在机器 A 上可作被控、在机器 B 上可作控制。

### 2.4 客户端目录结构（建议）

```
src/
 ├── controller/    # 控制端逻辑
 ├── agent/         # 被控端逻辑
 ├── webrtc/        # PeerConnection、DataChannel、ICE
 ├── signal/        # WebSocket 信令客户端
 └── ui/            # 界面
```

### 2.5 运行模式

| 模式 | 行为 |
|------|------|
| **Controller** | 输入控制码 → 建立 WebRTC 会话 → 显示远程画面 → 发送控制与（可选）接收状态 |
| **Agent** | 生成控制码 → 注册信令 → 等待连接 → 推送屏幕流 → 执行远端输入 |

### 2.6 与 AI 自动化逻辑架构（宿主应用内）

```
┌─────────────────────────────────────────────────────────────┐
│                     Electron 宿主应用                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐   │
│  │ 远程画面渲染  │  │ 任务/配置面板 │  │ 日志 / 报告预览   │   │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘   │
│         │                 │                    │            │
│         ▼                 ▼                    ▼            │
│  ┌────────────────────────────────────────────────────────┐  │
│  │         Runtime：任务循环 / 状态机 / 超时与停止          │  │
│  └────────────┬─────────────────────────┬────────────────┘  │
│               │                         │                    │
│               ▼                         ▼                    │
│  ┌────────────────────────┐  ┌────────────────────────────┐ │
│  │ WebRTC 控制与帧采样     │  │ Vision / LLM Adapter       │ │
│  │ DataChannel 输入 · 截图  │  │ 规划步骤 · 成功判定 · 报告   │ │
│  └───────────┬────────────┘  └────────────────────────────┘ │
└──────────────┼──────────────────────────────────────────────┘
               │
               ▼
        WebRTC P2P（对端 Agent）
```

**原则**：画面来自 **WebRTC 视频轨**（及/或从渲染管线获取与坐标系一致的帧用于 AI）；控制指令经 **DataChannel** 下发；信令仅做会话建立，不做视频中继（TURN 仅作 ICE 兜底路径）。

## 3. 功能需求

### 3.1 统一桌面客户端（被控端与控制端同一程序）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-APP-01 | 发布物为**单一桌面应用**（同一安装包/同一可执行产物），**被控端与控制端用户均安装该程序** | P0 |
| FR-APP-02 | 应用提供**明确入口**以进入 **Agent（被控）** 或 **Controller（控制）** 模式（主界面按钮、菜单或首次启动向导） | P0 |
| FR-APP-03 | 在**未建立远程会话**或产品定义的安全状态下，允许用户**切换模式**；切换时释放信令与 WebRTC 资源并给出提示 | P0 |
| FR-APP-04 | 可选支持**启动参数**指定默认模式（如 `--mode=agent`），便于固定机位只做被控 | P1 |
| FR-APP-05 | 关于「同一台电脑同时当控制端又当被控端」：不作为 MVP 必选项；若实现，需在 UI 与文档中说明限制（端口、权限、自连场景） | P2 |

### 3.2 控制码系统

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-CODE-01 | 被控端（Agent）启动后生成**唯一控制码**（示例形态：`83921122`） | P0 |
| FR-CODE-02 | 控制端输入控制码向信令服务发起匹配，建立对应设备的信令通道 | P0 |
| FR-CODE-03 | 控制码作为会话内**设备标识**与信令路由键；可与 `deviceId` 绑定 | P0 |
| FR-CODE-04 | 控制码**随机生成**；可配置**过期时间**；可绑定本机设备信息（扩展） | P1 |

信令侧数据结构示例（实现可对字段扩展）：

```json
{
  "code": "83921122",
  "deviceId": "device_001",
  "ws": "socket"
}
```

### 3.3 局域网优先连接（MVP）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-NET-01 | **优先使用局域网候选**（如 host 类型或内网 IP）完成 P2P，降低延迟 | P0 |
| FR-NET-02 | MVP 阶段在**同网段**场景下可不依赖 TURN；ICE 收集策略允许仅局域网可达 | P0 |
| FR-NET-03 | 跨网或直连失败时，回退到 **STUN**，再必要时 **TURN**（见第 5 章） | P1 |

### 3.4 WebRTC 连接与媒体

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-WEB-01 | 建立 **RTCPeerConnection**，完成 **offer/answer** 与 **ICE** 交换（经信令） | P0 |
| FR-WEB-02 | **视频轨**：被控端桌面/屏幕采集 → 编码 → 发送至控制端渲染 | P0 |
| FR-WEB-03 | 预留**双向音视频**扩展能力（额外轨或再协商） | P2 |
| FR-WEB-04 | **DataChannel**：至少一条用于**控制指令**（鼠标/键盘）；可选 **status-channel** 同步状态 | P0 |

默认 ICE 配置示例（可与产品配置合并）：

```js
iceServers: [
  { urls: "stun:stun.l.google.com:19302" }
]
```

### 3.5 鼠标键盘控制（经 DataChannel）

所有操作作用于被控端坐标系，与远程画面分辨率一致。

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-ACT-01 | **鼠标移动**到指定像素坐标 | P0 |
| FR-ACT-02 | **点击**：左键单击；可选双击、右键（按 OS/API 能力） | P0 |
| FR-ACT-03 | **拖拽**：按下 → 移动 → 释放 | P1 |
| FR-ACT-04 | **文本输入**：ASCII 及常用中文路径（输入法限制见非功能） | P0 |
| FR-ACT-05 | **按键**：Enter、Tab、Esc、方向键、快捷键组合（分阶段） | P0 |
| FR-ACT-06 | **滚动**：纵向/横向 | P0 |
| FR-ACT-07 | **等待**：固定毫秒；或与视觉轮询配合的「画面稳定」策略 | P0 |
| FR-ACT-08 | 每步可选防抖延迟，避免动画未完成导致误判 | P1 |

**通信方式**：上述控制指令经 **WebRTC DataChannel** 传输（二进制或 JSON 由实现约定 Schema）。

### 3.6 双向控制（扩展）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-BI-01 | 支持会话模型：**A 控制 B**；未来 **B 控制 A** 对称能力 | P2 |
| FR-BI-02 | **多人同时控制**、**Control Session** 与**控制权锁（Lock）** | P3 |

### 3.7 设备管理

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-DEV-01 | **设备列表**、**在线状态** | P1 |
| FR-DEV-02 | **最近连接记录**（控制码/别名/时间） | P1 |

### 3.8 图片识别 / 视觉 AI 对接（与 WebRTC 画面）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-AI-01 | 视觉分析输入：与执行坐标系一致的**帧/截图**（PNG/JPEG Base64 或 Buffer）、任务描述、可选历史摘要 | P0 |
| FR-AI-02 | 视觉分析输出：**结构化动作列表**（JSON Schema），禁止仅依赖自由文本作为机器解析通道 | P0 |
| FR-AI-03 | **多提供商适配**：配置切换 API Key、Endpoint、模型名 | P0 |
| FR-AI-04 | AI 调用失败时**可配置重试**与降级提示 | P0 |
| FR-AI-05 | Prompt 模板可配置 | P1 |

动作 JSON 示例（与实现 Schema 对齐即可）：

```json
{
  "steps": [
    { "action": "click", "x": 820, "y": 480, "target": "登录按钮" },
    { "action": "input", "value": "admin@example.com" },
    { "action": "scroll", "deltaY": -400, "x": 600, "y": 400 },
    { "action": "wait", "ms": 800 }
  ],
  "reasoning_trace": "可选，仅供日志"
}
```

### 3.9 Agent 执行循环与结果判断

| ID | 需求描述 | 优先级 |
|----|----------|--------|

| FR-LOOP-02 | **最大轮次**、**单轮最大步数**、**总超时** | P0 |
| FR-LOOP-03 | 单步结果判断（可选） | P1 |
| FR-LOOP-04 | 任务级成功判定（自然语言或结构化验收） | P0 |
| FR-LOOP-05 | 用户可随时 **停止**，释放锁定并记录步骤 | P0 |

### 3.11 日志与可观测性

## 4. 信令服务器设计

### 4.1 核心职责

- 控制码注册与路由
- WebRTC **offer/answer** 转发
- **ICE candidate** 转发

### 4.2 实现要点

- **Node.js + WebSocket** 长连接
- 房间或映射关系：`code` ↔ 对端 `socket`/会话
- 不转发媒体流（与 TURN 角色区分）

---

## 5. 网络策略

### 5.1 连接优先级

```
1. 局域网直连（优先）
2. STUN NAT 穿透
3. TURN 中继（兜底）
```

### 5.2 STUN（默认）

- 可使用 Google 公共 STUN：`stun:stun.l.google.com:19302`
- 无需自建即可满足多数 NAT 探测场景

### 5.3 TURN（后期扩展）

- 推荐使用 **coturn** 等自建 TURN
- 解决对称 NAT、企业防火墙等无法 P2P 的情况

## 8. 安全设计

### 8.1 控制码安全

- 随机、足够熵；防枚举速率限制（信令侧）
- 可选过期时间与设备绑定

### 8.2 通用安全

- 凭证与 API Key 不落日志明文
- 含敏感信息的调试截图需提示与访问控制

## 10. 产品演进路线

| 阶段 | 内容 |
|------|------|
| **Phase 1（MVP）** | 局域网 WebRTC、控制码、**同一 Electron 应用**内 Agent/Controller **双模式**、DataChannel 控制、基础设备标识 |
| **Phase 2** | 信令服务标准化部署、跨网段连接与连接记录 |
| **Phase 3** | STUN/TURN（含 coturn）、企业网络穿透 |
| **Phase 4** | AI 自动控制强化、多设备协同、控制权与会话高级策略 |

与自动化测试能力交叉时：**Phase 1** 可同时交付「手动控制 + 单提供商视觉闭环」的最小 AI 路径（见分阶段交付）。

### 10.1 分阶段交付（自动化与测试）

- **阶段 A — MVP**：**同一安装包**在两端分别作控制/被控 + WebRTC 连接 + 控制码 + 截图/取帧 + 点击 + 输入 + 等待 + 基础滚动 + 单一视觉 AI + 日志与超时  
- **阶段 B — 测试化**：验收描述、结构化失败原因、Markdown/HTML/JSON 报告  
- **阶段 C — 工程化**：多视觉提供商、Prompt 管理、CI 命令行与 exit code

## 11. 验收标准（建议）

1. **统一客户端**：同一安装包在**机器 A** 以被控模式生成控制码、在**机器 B** 以控制模式输入控制码并完成一次完整会话；两端为**同一产品**（非两个独立安装包名称）。  
2. **连接**：控制端输入控制码后，能在局域网内建立 WebRTC，画面与远端桌面一致且可重复连接/断开。  
3. **信令**：offer/answer 与 ICE 经信令正确转发，异常时有明确提示。  
4. **控制**：至少完成「打开应用/浏览器 → 点击可见区域 → 滚动 → 输入文本」且指令经 DataChannel 生效。  
5. **AI 闭环**（若本迭代包含）：截图/取帧 → JSON 动作执行 → 任务级 passed/failed 与日志一致。  
6. **报告**（阶段 B）：同一次运行可生成 JSON + 可读报告，时间线与步骤对应。

## 12. 技术选型总结

| 模块 | 技术 |
|------|------|
| 客户端 | Electron，**单一安装包**内置 Agent + Controller |
| 实时通信 | WebRTC（视频轨 + DataChannel） |
| 信令 | Node.js + WebSocket |
| NAT 穿透 | STUN；TURN（coturn，后期） |
| 控制通道 | WebRTC DataChannel |
| **明确不采用** | **VNC / RFB** 作为远程桌面传输协议 |

|------|------|
| WebRTC | 浏览器与客户端常用的实时通信框架，含 P2P、ICE、SRTP、DataChannel |
| 控制码 | 短码或数字串，用于在信令层匹配被控端会话 |
| Agent / Controller | 被控端模式与控制端模式；**均在同一桌面应用内**切换或入口选择 |
| 统一桌面客户端 | 同一可执行/安装产物，供控制端用户与被控端用户安装使用 |
| DataChannel | WebRTC 内低延迟数据通道，本 PRD 中承载控制消息 |
| STUN/TURN | ICE 辅助：NAT 探测与中继 |
| 视觉 AI | 输入图像、输出理解或结构化指令的多模态或专用服务 |

## 15. 结论

本系统采用「**轻信令 + P2P 实时通信**」架构：**不使用 VNC**，以 **WebRTC** 传输屏幕与 **DataChannel** 传输控制；**同一 Electron 桌面程序**供被控端与控制端安装使用，通过模式区分角色；**控制码**降低配对成本；**局域网优先**便于 MVP；**STUN/TURN** 平滑扩展至公网与企业网络；上层 **AI 视觉驱动自动化** 与数据通道天然契合，便于后续强化自动控制与多设备协同。
